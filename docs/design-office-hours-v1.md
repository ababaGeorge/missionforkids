## Problem Statement

Parents spend energy nagging kids about daily tasks (brushing teeth, cleaning desk, homework). Kids have no intrinsic motivation system for self-care and habit-building. The current solution is verbal reminders or physical sticker charts — both require constant parent involvement and lose novelty quickly.

Mission for Kids creates a gamified loop: parents define tasks → kids complete them with photo evidence → AI + parent review → points earned → points redeemed for real rewards. The emotional core is two-sided: kids feel autonomy and the dopamine of earning/spending, parents get the relief of self-driven children.

## What Makes This Cool

1. **The complete reward loop.** Unlike most habit trackers that stop at "check the box," kids earn a currency they actually spend. The reward store makes points tangible — ice cream, game time, pick a movie. This is what makes a 6-year-old care.

2. **AI photo review.** The app judges "did you do it?" from a photo — not quality, just completion. A clean desk looks clean. An open piano book with sheet music means practice happened. This removes the parent from the nagging loop while keeping them in the trust loop (48hr override window).

3. **The long-term vision.** Every completed task is a timestamped, photo-evidenced record of a child's growth. Today it's a chore app. Tomorrow it could be a verifiable learning portfolio — team challenges, self-directed learning, potentially school-recognized achievement records.

## Constraints

- Building for own family first — real users from day one
- iOS + Android (React Native / Expo)
- Firebase backend (Auth + Firestore + Storage + Cloud Functions)
- Must work for ages 6+ (simple UI, minimal text, large touch targets)
- 繁體中文 + English (i18n from day one)
- Must comply with COPPA / GDPR-K (see Compliance section below)
- AI review = "did you do it?" only (no quality judgment in v1)
- No punishment/deduction mechanics — positive reinforcement only

## Premises

1. The core loop requires all 5 steps: task → photo submit → review → points → redeem. Without ALL five, the product doesn't work.
2. AI review is included in the MVP as the key differentiator, but the system degrades gracefully to manual parent review when AI is unavailable or uncertain. The loop works with or without AI — AI makes it better, not possible.
3. The original planning docs define 12+ collections. The MVP trims this to 10 collections (see table below), deferring achievements, userStats, and userAchievements to post-MVP.
4. The school portfolio vision should NOT shape MVP architecture. Build the chore app first.
5. The family currently uses verbal reminders. The pain is real enough to use even a basic version daily.

## Approaches Considered

### Approach A: "Weekend MVP" — Minimal Core Loop
Single-family app with tasks, manual parent review, points, and a simple reward list. No AI, no recurring tasks. ~5 Firestore collections.
- Effort: S (human: ~1 week / CC: ~2-3 hours)
- Risk: Low
- Pros: Family can use it fast; forces core loop validation
- Cons: No AI review; no recurring tasks; bare-bones

### Approach B: "Full Core Loop" — Complete MVP as Designed
Tasks (one-time + recurring), semi-auto review (AI + parent 48hr window), points wallet, reward store with order flow. 10 Firestore collections.
- Effort: M (human: ~3-4 weeks / CC: ~1-2 days)
- Risk: Medium (AI integration adds complexity)
- Pros: Complete experience matching the planning docs; AI review is the differentiator
- Cons: More upfront work; AI prompt tuning needed

### Approach C: "PWA" — Ship Without App Store
Same scope as B but as a Progressive Web App (Next.js + Firebase). No App Store review process.
- Effort: M (human: ~2-3 weeks / CC: ~1-2 days)
- Risk: Medium (iOS PWA limitations)
- Pros: Ship instantly; one codebase; easy iteration
- Cons: iOS limitations; doesn't feel native; no App Store presence

## Recommended Approach

**Approach B: Full Core Loop.** The planning docs are already at this level of detail — data models, state machines, review logic are all designed. Building less would waste the thinking already done. AI review is the differentiator from PointUp/OurHome/Habitica.

### Key Technical Decisions

1. **Framework:** React Native (Expo) — broader ecosystem, easier to find help, Expo simplifies build/deploy
2. **Backend:** Firebase (Auth + Firestore + Storage + Cloud Functions)
3. **AI Review:** Cloud Function triggered on submission → sends photo to Claude Vision → returns pass/fail/uncertain with confidence score
4. **Task Scheduling:** Cloud Functions with Cloud Scheduler for recurring task instance creation and auto-approve countdown
5. **State Management:** React Context or Zustand — keep it simple for MVP
6. **Photo Storage:** Firebase Storage with per-family security rules

### Review Modes

The MVP supports two review modes, configurable per task:

| Mode | Description | When to use |
|---|---|---|
| **Semi-auto** (default) | AI reviews first → parent has 48hr override window → auto-approve if parent doesn't act (AI-pass only) | Most tasks — reduces parent workload |
| **Manual** | No AI involvement. Submission goes directly to parent queue. No auto-approve timer. Parent must explicitly approve. | Tasks where photo evidence is ambiguous (e.g., "be kind to your sister") |

### Retry Policies (AI Rejection vs. Parent Rejection)

| Scenario | Max retries | What child sees | After max retries |
|---|---|---|---|
| **AI rejects photo** | 5 per task instance | Rejection reason list + "Try again" | Routes to parent queue |
| **Parent rejects submission** | 1 resubmission allowed | "Can you try again?" + parent's optional message | Task instance marked missed (no punishment) |

**Daily AI cost cap:** Max 15 AI reviews per child per day across all tasks. After the cap, submissions route directly to parent review.

### AI Review Details

**Confidence thresholds:**

| AI confidence | Result | Behavior |
|---|---|---|
| > 0.8 | Pass | Routes to parent with "AI says: Looks good!" badge |
| 0.4 – 0.8 | Uncertain | Routes to parent with "AI uncertain" badge — no auto-approve |
| < 0.4 | Fail | Shows rejection reason to child, child can retry |

**Rejection reason list (fixed, shown with icons):**
- 📷 Photo is blurry — please retake
- 📐 Can't see the full area — please step back
- ❓ This doesn't seem to match the task — please check and retry

**Sample AI prompt template:**
```
You are reviewing a photo submitted by a child for a household task.
Task: "{task_title}"
Question: Has this task been completed based on the photo?
Respond with JSON: {"result": "pass"|"fail"|"uncertain", "confidence": 0.0-1.0, "reason": "brief explanation"}
Rules:
- Judge completion only, not quality
- If the photo is blurry or unclear, return "fail" with low confidence
- If you can see evidence the task was done, return "pass"
- If unsure, return "uncertain"
```

**AI cost estimate:** ~$0.01-0.04 per photo (Claude Vision). At 5-10 tasks/day for a family, capped at 15 AI reviews/day, this is ~$0.15-0.60/day or ~$5-18/month. Acceptable for MVP.

**AI unavailability fallback:** If the Vision API times out (>15s) or returns an error, the submission routes directly to parent review. The child sees "Sent to parent for review!" No AI badge is shown to the parent.

**Slow AI UX:** If review takes >3 seconds, show a playful loading animation ("AI is looking at your photo..."). If >15 seconds, auto-route to parent and show "Sent to parent!"

### Photo Upload Handling

- **Network failure:** Retry upload 3 times with exponential backoff (1s, 3s, 9s)
- **Persistent failure:** Queue photo locally and retry when network is restored
- **Child-facing message during retry:** "Uploading your photo..." (spinner)
- **Child-facing message on failure:** "Photo saved! We'll send it when you're back online."

### Rejection Flow (Child Experience)

When a parent rejects a submission:
1. Child receives a push notification: "Dad/Mom wants you to try again!"
2. Child sees the task back in their task list with a "Try again" badge
3. Child can resubmit ONE more time with a new photo
4. If rejected a second time, the task instance is marked as missed for this period (no punishment, no deduction — just moves to next period for recurring tasks)
5. No negative messaging — rejection message from parent is optional free-text, defaulting to "Can you try again?"

### Core Flows

**Child flow:**
1. Open app → see points banner + today's tasks
2. Tap task → camera opens → take photo → submit
3. AI reviews in background (loading animation if >3s)
4. AI pass → child sees "Looks good! Waiting for parent" (auto-approves 48hr after 23:59 notification if parent doesn't act)
5. AI uncertain → child sees "Sent to parent for review!"
6. AI fail → child sees rejection reason list + "Try again" button
7. Points appear in wallet → browse reward store → redeem

**Parent flow:**
1. Daily 23:59 notification → "3 items to review"
2. See AI verdict for each (pass/uncertain/routed from AI failure)
3. Approve or reject each (optional message on reject)
4. 48hr auto-approve for AI-pass items only. AI-uncertain/fail items stay until parent acts.
5. Manage tasks (create, edit, delete)
6. Manage reward store items

**Recurring task flow:**
1. Parent creates task with frequency (daily/weekly/monthly)
2. System creates TaskInstance for current period
3. On period end: if submitted → wait for review. If not submitted → grace period (default 2 days, configurable per task and per family) → marked missed
4. New TaskInstance auto-created for next period

**Grace period:** Default 2 days after period end. Configurable at family level (applies to all tasks) and overridable per task. During grace period, child can still submit. After grace period, instance is marked missed and deleted.

### Point Transactions (Atomic Operations)

All point-affecting operations use Firestore transactions inside Cloud Functions:

| Operation | Atomic steps |
|---|---|
| **Task approved** | Check taskInstance status → update to "approved" → increment pointWallet.balance → create pointTransaction (type: task_completion) |
| **Reward redeemed** | Check pointWallet.balance >= cost → decrement balance → create pointTransaction (type: reward_order, negative delta) → create rewardOrder |
| **Reward cancelled** | Check rewardOrder.status == pending → update to cancelled → increment pointWallet.balance → create pointTransaction (type: reward_refund) |
| **Manual grant** | Increment pointWallet.balance → create pointTransaction (type: parent_grant) |

### Family Invitation Flow

Parent invites a child via **in-app invite code:**
1. Parent taps "Add family member" → generates a 6-digit alphanumeric code (valid 24hr)
2. Code is stored in a `familyInvitations` subcollection under the family document
3. Child (or child's device) enters the code during onboarding → creates FamilyMembership with status "active"
4. Code is single-use and expires after 24 hours

For MVP, this is the simplest approach. QR codes or deep links can be added later.

### Firestore Security Rules (Outline)

| Collection | Parent Read | Parent Write | Child Read | Child Write |
|---|---|---|---|---|
| users | Own doc | Own doc | Own doc | Own doc (display name, avatar only) |
| families | Own families | Create only | Own families | No |
| familyMemberships | Own family | Own family (invite/remove) | Own membership | No |
| tasks | Own family | Own family (CRUD) | Own family | No |
| taskInstances | Own family | Own family | Own (assigned to self) | No |
| taskSubmissions | Own family | Approve/reject own family | Own submissions | Create (own instances only) |
| pointWallets | Own family | No direct write (Cloud Functions only) | Own wallet | No |
| pointTransactions | Own family | Create (manual grant only) | Own transactions | No |
| rewardItems | Own family | Own family (CRUD) | Own family | No |
| rewardOrders | Own family | Update status | Own orders | Create (own wallet sufficient balance) |

All writes to pointWallets happen through Cloud Functions to prevent balance manipulation.

### COPPA / GDPR-K Compliance

| Requirement | Implementation |
|---|---|
| Parental consent | Child accounts can only be created by a parent via family invitation. Parent must have a verified account (Apple/Google Sign-In). |
| Data minimization | Collect only: display name, avatar, birthday (child), auth provider ID. No address, no school, no location. |
| Photo data | Stored in Firebase Storage under family-scoped paths. Photos are used only for task review. No third-party sharing beyond the Vision API call. |
| Data retention | Photos retained for 90 days after task instance closes, then auto-deleted via Cloud Function. Point transaction history retained indefinitely (no photos). |
| Account deletion | Parent can delete child's account → all personal data, photos, and wallet data are permanently deleted. Point transaction notes are anonymized (user ID replaced with "deleted"). |
| Age gating | App requires family invitation to create a child account. No self-registration for children. |
| Push notifications | Request notification permission on first launch. If denied, fall back to in-app badge count on the Review/Tasks tab. Children's notification content is generic ("You have a new update!") — no task details in notification body. |
| Privacy policy | Required before App Store submission. Must disclose Vision API usage for photo review. |

### Wishlist Feature (願望清單)

Children can favorite rewards with a heart button (♡) on each reward card. Tapping ♡ → ♥ (filled, with a brief scale animation). The child's wishlist is visible on:
- **Child side:** "Me" tab shows a "My Wishes" section with wishlisted rewards and progress toward each ("★ 80/100")
- **Parent side:** "Rewards" tab shows a "Kids' Wishes" section — each child's avatar + their wishlisted items. This informs parents what rewards to create or prioritize.

**Future vision:** When vendor products are available, the wishlist becomes a savings goal system — children pick targets and see progress. This transforms passive browsing into active goal-setting, which is the motivation engine for ages 8-12.

### Vendor Currency Model (RESOLVED)

Vendors set real prices ($$), but **prices are only visible to parents.** Parents decide how many stars (★) a vendor product is worth — no fixed exchange rate, no conversion formula. The child's world contains only stars; they never see real currency amounts. Parents pay real money for the final purchase.

**Wishlist behavior:**
- Parent has set a star price → child sees "★ 80/100" progress toward the reward
- Parent has NOT set a star price → child sees "♡ Added to wishlist" with no star number
- This display logic is an unresolved minor decision (see Unresolved Design Decisions table)

**Architecture implication:** The reward store needs two layers:
1. **Star layer (child-facing):** All rewards show star prices. Child redeems with stars.
2. **Payment layer (parent-facing):** Vendor products show real prices. Parent completes purchase via Stripe/IAP.

For family-created rewards, there is no payment layer — stars are the only currency.

### Quiet Hours (靜音時段)

Parent setting: configurable time range (e.g., 21:00–07:00). During quiet hours, the app forces silent mode and uses haptic feedback only, regardless of device sound settings. Default: off (respects device mode).

### App Usage Time Management

The plan does NOT build usage time controls. Instead, the Settings page includes a guidance section: "Use Apple Screen Time (iOS) or Google Family Link (Android) to manage app usage time." Avoids reinventing the wheel and reduces parent friction of managing two systems.

### Profile Switching (Multi-Child Households)

On the child "Me" tab, a profile switcher shows all child avatars in the family.

- **No PIN set:** Tap avatar → switch immediately
- **PIN set:** Tap avatar → number pad (4-digit PIN) → switch on correct entry

PIN is set by the child themselves (in "Me" tab settings). Parent can view PINs but cannot modify them. This gives children privacy and prevents younger siblings from accidentally modifying tasks or spending stars.

3. If parent has created tasks → task list with pulsing glow on first task

### AI Review Feedback — Visual-First (Child-Facing)

Replace text-heavy rejection reasons with visual-first feedback for children:
- **Blurry photo:** Large camera character illustration + "再試一次！拍清楚一點 📸 / Try again! Make it clearer"
- **Can't see full area:** Large zoom-out icon + "退後一點，讓我看到全部！/ Step back so I can see everything!"
- **Doesn't match task:** Large question mark illustration + "好像不太對，再確認一下？/ Hmm, that doesn't look right. Check and try again?"

Each has ONE large illustration, ONE short sentence, and ONE big colorful "Try Again" button. Coaching tone, not correcting tone.

### Pending Stars (★ → ☆)

After AI pass, before parent approval: task shows "☆ 10 stars pending" (hollow/ghost stars) in the child's star area. When parent approves, hollow stars fill in (☆→★) with the celebration animation. This bridges the emotional gap during the waiting period (potentially 16-64 hours between submission and parent action).

## Reviewer Concerns

The following items were flagged during adversarial review but not resolved in the design doc. Address these during implementation:

1. **`familyInvitations` subcollection missing from collections table and security rules.** Add to both, or document as a subcollection under `families` with its own access rules.
2. **Missed task instance retention unclear.** Are missed instances soft-deleted (status: missed) or hard-deleted? Define retention policy.
3. **Reward order lifecycle undefined.** Define rewardOrder statuses (pending → fulfilled → completed/cancelled) and parent fulfillment action.
4. **Timezone handling unspecified.** All daily boundaries (23:59 notifications, period end, grace period) need a timezone source. Recommendation: per-family timezone stored in `families` document.
5. **48hr auto-approve effective window is 48-64hr** depending on submission time (child submits at 08:00, parent notified at 23:59, 48hr starts then). Document this explicitly so it's not mistaken for a bug.
6. **AI cost worst case for multi-child families.** With 15 AI reviews/day/child cap, a 3-child family could hit ~$54/month worst case. Confirm the cap is per-child.
7. **Photo retention cleanup function** not in milestone plan. Add to Week 3 or 4.
8. **Concurrent approval race condition.** Firestore transaction handles this via optimistic locking — note for the engineer implementing the approval Cloud Function.
