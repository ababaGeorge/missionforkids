# Mission for Kids — 人工實機測試計畫

> 版本：對應 `main`（2026-07-06 development build）
> 對象：測試員（cowork）｜產出：逐項 Pass/Fail + 缺陷記錄
> 目的：驗證家長端與小孩端**每個功能與按鈕都如實對應**、**正確寫入資料庫並讀回**、**兩角色間資料互動正確**，並找出 UI/導覽的阻擋與死迴圈。

---

## 0. 這份怎麼用（測試員必讀）

1. **一項一項照做**，在每列「結果」欄填 ✅通過 / ❌失敗 / ⚠️怪怪的，失敗必寫「實際看到什麼」。
2. **重點是「資料真的存進去且讀得回來」**。你沒有資料庫後台，所以用這三招驗證，任一過關就算寫入成功：
   - **跨角色一致**：家長做的事，小孩那台/那個帳號看得到（反之亦然）。← 最重要
   - **重開仍在**：把 App 完全關掉（上滑殺掉）再打開，資料還在 = 有存進資料庫（不是只存在畫面）。
   - **（選用，交給開發者）** Firebase Console 對應集合可覆核，見附錄 B。
3. **先看第 5 節「已知缺口」**——那些是還沒做的功能，看到不動是正常的，**別當 bug 報**。
4. **第 4 節「重點盯防」是我們已經懷疑有問題的地方**，請特別用力測、如實回報。
5. 缺陷用第 6 節模板記錄（附截圖 / 螢幕錄影最好）。

## 附錄 A：資料如何在資料庫落地（給想深入的人）

- **點數一律由後端 Cloud Functions 寫入**，App 端只改「狀態旗標」觸發：完成任務核准→`onTaskInstanceApproved` 發點；兌換下單→`onRewardOrderCreated` 扣點；取消/婉拒→`onRewardOrderCancelledOrRejected` 退點；家長手動→`grantPoints`。
- 錢包是「一小孩一錢包」：`pointWallets/{familyId}_{childId}`。所以「跨角色一致 + 重開仍在」就是最好的寫入驗證。
