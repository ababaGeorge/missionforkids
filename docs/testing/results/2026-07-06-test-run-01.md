## 0. 執行摘要（先看這段）

**核心閉環結論：能完整走通。** 建任務 → 小孩完成 → 家長審核發點 → 小孩兌換扣點 → 家長核准 → 交付 → 小孩「我拿到了」這條主鏈路全程可跑通，**點數計算全程正確、跨角色資料即時同步、持久化正確**（多次登出/登入重新從 Firestore 讀回都一致）。

**發現的阻斷/重要問題：**
- 🔴 **BUG-01（= 重點①/U1 已證實）**：小孩兌換訂單按「我拿到了」後，底部兩顆按鈕（回到主頁 / 回到任務）**都變成點了沒反應**，只能靠右上角 `✕` 離開。訂單狀態本身有正確完成、點數也正確（不再扣），但對兒童使用者這是「看起來壞掉」的最後一步。
- 🟠 **BUG-02（任務指派 / 重複 instance）**：新建任務時已選指派對象，建立後任務卡卻顯示「未指派・0/0」（重新登入才更正為「小安・0/3」）；而且 1 次建立 + 2 次編輯儲存 → **長出 3 個 instance**，小孩端看到**重複的同一個任務**。編輯 modal 也**不預填**現有指派對象。
- 🟡 其餘：商城鎖死無卡片說明（⑥ 已證實）、取消兌換退點為 async 有數秒延遲、家長通知未即時反映新事件、審核 sheet「兌換前」餘額顯示可能不符實際。

**阻斷數：** 1 個需優先修的體驗阻斷（BUG-01，非資料錯，可用 ✕ 脫困）。**資料錯誤：0 個**（點數帳全對）。

**建議修復順序：** BUG-01（我拿到了死鎖）→ BUG-02（指派顯示 + 重複 instance）→ 其餘 🟡。

## ⚠️ 本次測試的環境限制（影響覆蓋率，請務必知悉）

1. **模擬器文字/數字輸入失效（最大限制）**：模擬器連著硬體鍵盤且疑似掛著中文輸入法，直接打字（type/key）**打不進 App 的輸入欄**，還會誤觸系統的重音選字彈窗、選字候選列、貼上選單。
   - **繞過法**：數字欄位用 `Cmd+K` 叫出**軟體數字鍵盤**，用點擊輸入 → 有效（發點金額就是這樣測成的）。
   - **仍受阻**：任何需要**自由中文/文字**輸入的案例（自訂任務名、拒絕留言、家長註冊、Email 邀請填寫）無法穩定輸入 → 這些案例**未能測**或改用「範本 chip / dev 假提交」等免打字路徑。任務名一律用「快速範本」chip 帶入。
2. **截圖無法存檔**：本 session 的 `save_to_disk` 無效，**無法產出 `screenshots/` 檔案**。以下所有結果以文字詳述實際畫面觀察替代截圖。
3. **iOS 系統通知 banner 偶爾蓋住模擬器**攔截點擊（顯示 UserNotificationCenter 在前景），需把 Simulator 拉回前景才能繼續 → 拖慢流程，非 App 問題。
4. 因單一模擬器 + 登入切換成本高，跨角色測試採「登出→登入另一帳號」批次進行；即時雙裝置互動（如審核當下的慶祝動畫）未能觀察。

## 6. 整體結論

**核心閉環：可完整走通，且點數帳全對、跨角色即時同步、持久化正確。** 這是最重要的結論——產品主幹是通的。

**阻斷級問題僅 1 個（BUG-01「我拿到了」死鎖）**，且是前端按鈕狀態問題、非資料錯，可用 ✕ 脫困；但它正好卡在獎勵閉環的最後一步、面對兒童使用者，**建議最優先修**。

**次優先為 BUG-02（指派顯示延遲 + 重複 instance）**：牽涉「家長派任務給小孩」這條核心，雖然指派其實有寫入、小孩看得到，但「顯示未指派」會誤導家長重做、且會長出重複任務，值得儘快查 create/edit 的 instance 生成邏輯與卡片刷新。

其餘 🟡（商城鎖無說明、退點延遲、家長通知不同步、審核前餘額顯示）為體驗打磨，成本低、可批次處理。

**尚未驗證但重要**：重點② 對「受邀小孩」發點是否失敗——因環境無法建立受邀小孩而未測，請在可正常輸入的環境優先補測。

建議修復順序：**BUG-01 → BUG-02 → BUG-03/04/05 → 補測 ②/③ 與其餘受限項**。
