# Mission for Kids — UX 改善建議

> 版本：對應 `main`（2026-07-06）
> 來源：家長端／小孩端／跨角色三份程式碼盤點 + 登入畫面實機觀察
> 定位：這是**基於現有實作的 UX 診斷與建議**；測試員實測時再補充實際體感（見文末「測試員補充區」）。
> 分級：🔴 阻斷體驗（會卡住/誤導到做不下去）｜🟡 摩擦（能用但困惑/易錯）｜🟢 打磨（不影響功能）

---

## 一、🔴 阻斷型（優先修）

### U1. 小孩兌換訂單「我拿到了」後畫面卡死
- **問題**：按下「我拿到了」→ 狀態更新為 completed 後，底部兩顆按鈕都保持 disabled（送出中旗標沒重設、也沒導覽），小孩只能用手機系統手勢離開。
- **影響**：小孩（目標使用者是兒童）會以為「壞掉了」。這是完整兌換流程的**最後一步**，卡在這裡等於獎勵閉環斷掉。
- **建議**：完成後自動導回獎勵頁或顯示「已完成」＋一顆明確的「回主頁」鈕（比照「取消」成功後的處理）。

### U2. 對「受邀綁定小孩」發點可能直接報錯
- **問題**：發點用的是 user 文件 id，但後端用 membership id 查，受邀綁定的小孩兩者可能不一致 → 報「不是家庭成員」。
- **影響**：家長對真實邀請進來的小孩發點失敗，是核心功能斷裂。
- **建議**：發點入口統一改用 `membership.userId`（而非 user doc id），與後端一致。

## 二、🟡 回饋與清晰度（使用者看不懂發生什麼事）

### U3. 載入中顯示「空狀態」，像是失敗了
- **問題**：任務／禮物／審核頁在資料載入前直接顯示「還沒有任務」「都審完了 🎉」，沒有 loading 區別。
- **影響**：家長剛建完任務切過去，可能看到「還沒有任務」以為沒存成功，重複建立。
- **建議**：加骨架屏/loading 狀態，資料確定為空才顯示空狀態文案。

### U4. 送出類操作缺乏「進行中／成功」回饋
- **問題**：建任務、建禮物按下後沒有明確 in-flight／成功提示（發點有做防連點，其他沒有）。
- **影響**：使用者不確定成功與否 → 重複點（見 U5）。
- **建議**：所有寫入按鈕統一：按下即 disabled + loading，成功給輕量 toast/勾勾。

### U5. 建任務／建禮物可被連點建立重複
- **問題**：這兩個建立流程沒有送出中鎖，非同步寫入期間可再次點擊。
- **影響**：出現兩個一模一樣的任務/禮物。
- **建議**：加送出中鎖（比照發點的 `granting` + idempotency 作法）。

### U6. 扣點成功提示顯示「請求值」而非「實際扣除值」
- **問題**：小孩餘 5、扣 10，實際 clamp 到 0（扣 5），但提示仍寫「−10 ★」。
- **影響**：家長帳目認知錯亂。
- **建議**：提示顯示實際變動量（後端已知道真實 delta，回傳給前端顯示）。

### U7. 小孩商城「有進行中訂單就全鎖死」且無說明
- **問題**：只要有一筆未完成訂單，所有獎勵卡變灰不可點，卡片上沒有任何文字解釋。
- **影響**：小孩一頭霧水「為什麼都點不動」。
- **建議**：灰掉的卡片上加一句「先完成上一個兌換喔」，或把限制說明放在明顯處。

## 三、🟡 可發現性與一致性（找不到、兩套入口）

### U8. 刪除任務的主要手勢是「隱藏長按 800ms」
- **問題**：點卡片是編輯，刪除靠長按或不明顯的「⋯」。
- **影響**：使用者找不到怎麼刪任務。
- **建議**：把刪除放進編輯 modal 內（比照禮物的「刪除這個禮物」），或卡片右滑露出刪除。

### U9. 兌換訂單有「兩個入口、狀態不一致」
- **問題**：pending 訂單要去「審核」頁按「答應」；但「禮物頁→兌換紀錄」看同一筆卻沒有操作鈕，要等 approved 後才在禮物頁出現「已交付」。
- **影響**：家長在禮物頁看到待批的訂單卻不知道去哪批。
- **建議**：兌換相關操作集中在一處，或在禮物頁的 pending 訂單上放一個「去審核」捷徑。

### U10. 通知點擊不導覽 + 已讀不持久
- **問題**：點通知只在本機標已讀、不跳到對應任務/訂單；重開 App 已讀全部重置。
- **影響**：通知形同「看看就好」，無法點進去處理，且每次重開又全變未讀（吵）。
- **建議**：通知卡點擊導覽到對應審核/任務頁；已讀狀態寫回 Firestore（或本機持久化）。這也是未來接推播（B2）的前置。

### U11. Modal 點背景不關閉、部分送出無回饋
- **問題**：所有 modal 點背景只收鍵盤不關閉，要按「取消」；婉拒 sheet 在留言空白時「送出」靜默無反應（只是按鈕半透明 disabled）。
- **建議**：modal 背景點擊可關（或至少視覺上明確只能按鈕關）；disabled 送出鈕旁給一句「請先填寫留言」。

## 四、🟢 假資料與佔位（會誤導、顯得未完成）

### U12. 小孩「我的」頁大量假資料
- **問題**：連續天數恆 0、「讀書蟲」徽章恆顯示已取得、部分徽章永遠拿不到；長條圖才是真資料。
- **影響**：獎勵/成就感是這 App 的核心動機，假徽章會讓小孩困惑、削弱信任。
- **建議**：要嘛把徽章/連續天數接上真資料，要嘛先隱藏未實作的，別擺假的。

### U13. 設定頁多個「尚未開放」佔位
- **問題**：家長設定（語言/通知/審核方式/螢幕時間）、小孩設定（語言/家長協助）多為點了跳提示或死元件。
- **建議**：未實作的先隱藏，避免整頁看起來都不能用；「審核方式：手動」是寫死顯示，不反映各任務實際設定，宜移除或修正。

## 五、通知體驗（連動 B2 產品缺口）

現況：**沒有推播**，站內通知只在 App 開著且切到通知頁時即時出現，已讀不持久。對「取代嘮叨」這個產品定位而言，通知是關鍵機制。建議把「通知體驗」當一個完整題目做（見專案 B2）：
1. 推播：家長審核、小孩加點、兌換申請等事件即時推送對方。
2. 站內通知升級：可點擊導覽、已讀持久、未讀紅點。
3. 需先拍板一個產品問題：**小孩的通知要推到誰的裝置**（小孩自己的手機 vs 共用家長裝置）——這會決定整個推播路由。

---

## 六、建議修復順序（我的觀點，供參考）

1. **先修 U1、U2**（🔴 阻斷核心閉環的兩個）——這兩個會讓「兌換完成」和「發點給真小孩」直接壞掉。
2. **再處理 U3–U7**（回饋與重複建立）——低成本、高體感提升，且能消掉「以為失敗又重做」的資料髒亂。
3. **U8–U11**（可發現性/一致性）——中等成本，讓流程更順。
4. **U12–U13**（假資料/佔位）——快速清理，讓 App 看起來「完成度高」。
5. **通知體驗（B2）** 當獨立產品題目排程。

> 這些是從程式碼推斷的診斷。**實機測試會驗證哪些真的咬人、以及還有哪些沒被程式碼揭露的體感問題**——測試員請把體驗上的卡頓、困惑、不直覺補在下方。
