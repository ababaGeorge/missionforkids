## Problem Statement

Parents constantly nag kids to do chores and self-care tasks. Kids lack intrinsic motivation for repetitive daily routines. No existing app combines gamified task completion with AI photo verification and a family reward store in a single, child-friendly experience.

The builder is a parent solving their own family's problem — building something their own kids will use daily.

## What Makes This Cool

Two-sided delight:
- **Kids:** The dopamine of earning points and spending them on things they actually want. "I did 7 days of cleaning my desk, now I get to pick a movie tonight."
- **Parents:** The relief of not having to nag. The app does the reminding, the kid does the doing, the parent just sees the streak growing.

## Future Vision (not scoped in this document)

The long-term vision transforms this from a chore app into a **verifiable learning portfolio** — every task, challenge, and self-directed learning moment timestamped and evidenced with photos. The ultimate fantasy: a school-recognized record of childhood development. This is aspirational and has no implementation scope in this document.

## Constraints

- Testing with own family first — must be usable by real kids (6+ years old)
- iOS + Android (Expo handles both)
- Mandarin Chinese + English — all screens (parent and child views) are bilingual; zh-TW strings are owner-provided at build time. Child and parent can use different language settings.
- No punishment mechanics — positive reinforcement only
- COPPA/GDPR-K compliance required before App Store submission (see Open Questions). Family TestFlight testing uses parent-created accounts without the public-facing consent UI.
- **Offline scope:** Firestore offline persistence for task viewing and document reads/writes. **Photo submission requires connectivity** — Firebase Storage does not support offline upload queuing. Offline photo queuing is deferred unless family testing reveals it as a real pain point.

## Premises

1. Kids (6+) will proactively open the app and submit photo evidence without being nagged
2. The earn-spend loop sustains long-term motivation without punishment mechanics
3. AI photo review ("did you do it?") is technically feasible with sufficient accuracy using current vision models
4. Parents will invest time setting up tasks and reward items if the initial setup is guided
5. Firebase + Expo is sufficient for all MVP features (offline sync for reads, scheduling, AI review, photo storage)

## Landscape

### Competitors (chore/reward apps)
- **PointUp** — closest competitor. RPG quests, levels, badges, photo proof, ADHD-friendly. Deep gamification.
- **Habitica** — RPG-style habit tracker, popular but not family-focused. No parent-child relationship.
- **OurHome** — free family organizer with points and rewards. Simple but no AI, no photo verification.
- **ClassDojo** — classroom behavior tracking, not home-focused.
- **TaskieHusky, Levelty** — smaller players with basic chore tracking.

### Learning portfolio space
- **Seesaw** — 25M+ users, school-recognized, PreK-5th grade. School-initiated, not parent-initiated.
- **ClassDojo Student Stories** — student-led portfolios within classroom context.

### Gap
No existing app bridges home chore tracking to learning portfolio. The chore apps don't think about long-term records; the portfolio apps are school-initiated, not family-initiated.

## Approaches Considered

### Approach A: Lean Core Loop
One-time tasks only, manual review, points, simple reward store. Ships in 1-2 days.
- Completeness: 5/10

### Approach B: Full MVP Core Loop
Complete implementation per decision docs — recurring tasks, semi-auto AI review, full reward store with order flow. 1-2 weeks.
- Completeness: 8/10

### Approach C: Core + AI Demo (CHOSEN)
Lean core loop (one-time tasks, manual review, points, reward store) PLUS a standalone AI Review Playground. 3 days for core loop, 5 days total including AI Playground.
- Completeness: 7/10

## Recommended Approach

**Approach C: Core + AI Demo**

Rationale: Gets a working app into the family's hands within days while simultaneously testing the hardest technical unknown (AI photo review accuracy) in a low-risk way. If AI works well in the playground, promote it to real review mode. If it doesn't, the core app still works with manual review.

### What gets built

Timeline assumes CC (Claude Code) handles boilerplate generation and Firebase setup. Each feature is cut to minimum viable — happy path only for Phase 1.

**Phase 1 — Core Loop (Days 1-3):**

1. **Expo project setup:** TypeScript template, Firebase config, `react-i18next` for i18n (zh-TW + en)

2. **Firebase setup:** Firestore, Auth, Storage, Cloud Functions. **Firestore Security Rules enforcing family-scoped access are part of this step** — a child can only read/write data within their own family. This is not deferred.

3. **Auth:**
   - **Parent:** Sign in with Apple or Google via Firebase Auth. Creates and manages their own account.
   - **Child (under 13):** Parent creates a child profile within the family (display name, avatar, birthday). Parent generates an invite link or QR code. Child taps the link on their own device, which creates a Firebase Auth anonymous account upgraded with a long-lived token scoped to the child profile. The child device stays signed in — no Apple/Google ID required for children. Session is managed by the parent account (parent can revoke access).
   - **COPPA parental consent gate:** Deferred to pre-App Store submission. Family TestFlight testing uses parent-created accounts directly.

4. **Family creation + member invitation:** Parent creates family, invites other parents (via Apple/Google auth) or kids (via link/QR as described above).

5. **Task creation (one-time only):** Parent sets title, points, due date, assigns to specific kid(s).

6. **Task submission:** Kid taps "done" + takes photo. **Requires connectivity** for photo upload. On upload failure: show a toast "無法上傳照片，請檢查網路 / Photo upload failed, check your connection" with a retry button.

7. **Manual review — Pending Reviews screen:** Dedicated screen in parent dashboard showing each submission with photo thumbnail, task name, kid name, and approve/reject buttons.
   - **On approval:** Points credited to kid's wallet via server-side Firestore-triggered Cloud Function (not client-side — prevents spoofing). Kid sees a celebration animation.
   - **On rejection:** Kid sees a friendly message ("再試一次！" / "Try again!") with optional parent note. Submission status resets to allow resubmission with a new photo. Original photo is retained in submission history. Parent notes persist across resubmissions. **Resubmission limit: 3 attempts per task instance** — after 3 rejections, task is marked as incomplete (no penalty, no point deduction, consistent with positive-reinforcement-only constraint). Parent can manually override to approve at any time.

8. **Points:** Wallet per kid (scoped to family) — each child has their own balance within each family. **Point crediting is performed server-side** by a Firestore-triggered Cloud Function on `taskInstances` status change to `approved`. Client code cannot directly modify wallet balances.

9. **Reward store + fulfillment flow:** Parent creates items with point costs. Kid browses and redeems. State machine:
   - `pending` — kid redeems, points deducted immediately
   - `approved` — parent reviews and approves the order (confirms they will fulfill it)
   - `delivered` — parent marks as physically/virtually delivered to kid
   - `completed` — kid confirms receipt, OR auto-completes 72 hours after parent marks `delivered` if kid has not confirmed
   - `cancelled` — kid cancels before parent approves (points refunded immediately)
   - `rejected` — parent declines the order (points refunded immediately). E.g., item out of stock or kid already received equivalent.

**Phase 2 — AI Playground (Days 4-5):**

10. AI Review Playground screen: kid takes a photo, client shows an animated "AI 正在看... / AI is thinking..." state. Cloud Function calls OpenAI Vision API (typical latency 2-6 seconds). AI responds with "看起來你做到了！/ Looks like you did it!" or "嗯，我不太確定 — 再試一次？/ Hmm, I'm not sure — try again?". 10-second timeout — if exceeded, show fallback: "AI 想太久了，晚點再試 / AI took too long, try again later."

11. Results logged to Firestore for accuracy analysis — no impact on actual task flow.

**Phase 3 — Iterate (Week 2+):**

12. Add recurring tasks (daily/weekly/monthly) with Cloud Functions scheduling
13. Promote AI review to **semi-auto mode** if playground data shows >85% accuracy on real family photos. Semi-auto defined: AI auto-approves submissions where confidence > threshold (e.g., 0.9); low-confidence submissions fall through to manual parent review. Parent retains 48-hour override window on AI-approved items.
14. Achievement badges and streaks — uses deferred `userStats`, `achievements`, `userAchievements` collections from the existing data model docs.

### Tech stack

| Layer | Choice |
|-------|--------|
| Frontend | Expo (React Native) + TypeScript |
| Navigation | Expo Router (file-based routing) |
| State | React Context + Firestore real-time listeners. Listeners attached at family-session level (not per-screen), detached on sign-out. This is a known tradeoff — reduces per-screen complexity but requires careful lifecycle management on low-end Android devices. |
| Backend | Firebase (Firestore, Auth, Storage, Cloud Functions) |
| AI | OpenAI Vision API via Cloud Function (playground only in Phase 2). Alternative models (Google Cloud Vision, on-device ML) evaluated based on playground accuracy data before Phase 3 promotion. |
| Auth | Firebase Auth — Apple + Google for parents, anonymous auth with long-lived token for children |
| i18n | `react-i18next` (full pluralization, namespace support, React Native compatible) |
| Offline | Firestore offline persistence for document reads/writes. Photo uploads are online-only. |
| Security | Firestore Security Rules enforce family-scoped access. Cloud Functions handle all point mutations. |

### Data model (MVP subset)

From the existing decision docs, MVP uses these collections:
- `users` — parent/child accounts
- `families` — family units
- `familyMemberships` — user-family relationships with role and status
- `tasks` — task definitions (one-time only in Phase 1)
- `taskInstances` — per-kid task assignments
- `taskSubmissions` — photo submissions + review status (retains all photos across resubmissions)
- `pointWallets` — per-kid-per-family balance (one wallet per child per family, write-protected by Security Rules)
- `pointTransactions` — earn/spend ledger (written only by Cloud Functions)
- `rewardItems` — store items (family-scoped)
- `rewardOrders` — purchase orders: pending/approved/delivered/completed/cancelled/rejected

Deferred to Phase 3+: `userStats`, `achievements`, `userAchievements`, `Recurrence` logic.

## Open Questions

1. **AI model choice:** OpenAI Vision API is the starting point for the playground. Accuracy data from real family photos will determine whether to keep it, switch to Google Cloud Vision, or explore on-device ML before promoting to production review mode.
2. **Push notifications:** Expo Push Notifications or Firebase Cloud Messaging? Need to decide before recurring tasks (Phase 3).
3. **Photo storage costs:** Firebase Storage pricing at scale — may need compression or retention policy. Not a concern for single-family MVP.
4. **COPPA/GDPR-K compliance:** Non-trivial for under-13 users. Implementation approach needed before App Store submission: (a) verifiable parental consent gate during child account creation, (b) no behavioral advertising or third-party data sharing, (c) data deletion on parent request, (d) minimal data collection principle. Detailed compliance plan to be written before public distribution — not blocking family TestFlight testing.

## Success Criteria

1. Own family uses the app daily for 1 week straight
2. Kid completes at least 3 tasks per day without being reminded
3. Kid redeems at least 1 reward from the store
4. AI Playground correctly identifies "done" vs "not done" in >85% of test photos
5. Both parent and child find the app easier than verbal nagging

- You said "對父母來說是，我的孩子每天自動自發完成自理工作" — you're not building a chore tracker. You're building a parenting tool that makes nagging obsolete. That's a much bigger idea than most people in this space articulate.

- Your ultimate fantasy — "演變成學校認可的一種學習歷程" — reveals that you see this as infrastructure for childhood development, not just a utility app. That vision is what separates a side project from something that could matter.

## Reviewer Concerns

Remaining issues from 3 rounds of adversarial spec review (quality score: 7/10). These are implementation-level details to resolve during development:

1. **Cloud Function failure handling for point crediting** — need retry policy and dead-letter strategy
2. **Child session revocation UX** — what child device sees when parent revokes access
3. **Reward cancellation exploit** — kid could cancel-redeem-cancel in a loop; consider cancel cooldown
4. **AI Playground lacks task context** — Vision API prompt needs task description for meaningful accuracy
5. **Photo compression before upload** — need max dimension (e.g., 1024px) and file size limit (e.g., 2MB)
6. **Success criterion "without being reminded" is untestable** — replace with observable proxy metric
7. **Completeness gap acknowledgment** — Approach C at 7/10 should note recurring tasks are explicitly deferred
8. **Refund transactions** — explicitly confirm cancellation/rejection refunds go through Cloud Functions
9. **Child auth mechanism** — specify exact Firebase Custom Auth Token flow for anonymous child accounts
10. **Auto-complete trigger** — specify scheduled Cloud Function for 72-hour reward auto-complete
11. **AI confidence score** — OpenAI Vision API returns text, not numeric confidence; need proxy scoring approach
12. **Firestore listener lifecycle** — tear down on app background, re-establish on active for low-end Android
13. **Cloud Function cold start** — consider min-instances:1 during testing, or extend UX timeout for first call
14. **COPPA stub for TestFlight** — implement non-blocking consent UI before any App Store submission
